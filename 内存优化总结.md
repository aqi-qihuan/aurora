## ⚙️ 详细优化配置

### 1️⃣ SpringBoot JVM 极限优化

**docker-compose.yml 配置** (`/opt/aurora/docker-compose.yml`):

```yaml
command: >
  java -Xms64m -Xmx128m -XX:+UseSerialGC
      -XX:MetaspaceSize=64m -XX:MaxMetaspaceSize=128m
      -XX:ReservedCodeCacheSize=32m -Xss256k
      -XX:+UseCompressedOops
      -XX:+UseCompressedClassPointers
      -XX:+UseStringDeduplication
      -XX:+OptimizeStringConcat
      -XX:+AlwaysPreTouch
      -Djava.security.egd=file:/dev/./urandom
      -Dspring.jmx.enabled=false
      -Dspring.backgroundpreinitializer.ignore=true
      -jar /app/blog.jar
```

**优化说明**：

| 参数 | 说明 | 效果 |
|:-----|:-----|:-----|
| `-XX:+UseSerialGC` | 使用 SerialGC 替代 G1GC | 小堆内存下更高效 |
| `-Xms64m -Xmx128m` | 堆内存压缩至极限 | 节省大量内存 |
| `-XX:MetaspaceSize=64m` | 元空间压缩 | 减少内存占用 |
| `-XX:ReservedCodeCacheSize=32m` | 代码缓存压缩 | 降低内存消耗 |
| `-Xss256k` | 线程栈压缩 | 减少线程开销 |
| `-Dspring.jmx.enabled=false` | 禁用 JMX | 减少开销 |
| `-XX:+UseCompressedOops` | 压缩指针优化 | 减少内存使用 |
| `-XX:+AlwaysPreTouch` | 预分配内存页 | 减少运行时开销 |

> 📌 **当前实际内存**：356.4MiB（堆内存128MB + 元空间128MB + 代码缓存32MB + Native内存）

### 2️⃣ MySQL 内存优化

**docker-compose.yml 配置**:

```yaml
command: >
  --innodb_buffer_pool_size=96M
  --innodb_log_buffer_size=4M
  --innodb_flush_log_at_trx_commit=2
  --sync_binlog=100
  --max_connections=30
  --table_open_cache=100
  --thread_cache_size=5
  --performance_schema=OFF
  --skip-name-resolve
  --tmp_table_size=4M
  --max_heap_table_size=4M
  --innodb_flush_method=O_DSYNC
```

**优化说明**：

| 配置项 | 原始值 | 优化值 | 说明 |
|:-------|:------:|:------:|:-----|
| `max_connections` | 151 | 30 | 进一步降低 |
| `table_open_cache` | 4000 | 100 | 大幅压缩 |
| `innodb_buffer_pool_size` | 128M | 96M | 适度降低 |
| `performance_schema` | ON | OFF | 完全禁用 |
| `tmp_table_size` | 8M | 4M | 降低50% |

### 3️⃣ Redis 内存优化

**docker-compose.yml 配置**:

```yaml
command: >
  redis-server --requirepass aqi1015
  --maxmemory 24mb
  --maxmemory-policy allkeys-lru
  --save ""
  --appendonly no
  --maxclients 100
  --tcp-backlog 128
  --timeout 300
```

**优化说明**：

| 配置项 | 说明 |
|:-------|:-----|
| `--maxmemory 24mb` | 限制最大内存（从 32MB 降低） |
| `--maxmemory-policy allkeys-lru` | LRU 淘汰策略 |
| `--save ""` | 禁用 RDB 持久化 |
| `--appendonly no` | 禁用 AOF 持久化 |

### 4️⃣ Elasticsearch 内存优化

**docker-compose.yml 配置**:

```yaml
environment:
  discovery.type: single-node
  ES_JAVA_OPTS: "-Xms64m -Xmx160m -XX:+UseG1GC -XX:MaxGCPauseMillis=200"
  TZ: Asia/Shanghai
  bootstrap.memory_lock: "true"
  xpack.security.enabled: "false"
  indices.query.bool.max_clause_count: "1024"
  indices.queries.cache.size: "5%"
  indices.fielddata.cache.size: "0%"
  indices.memory.index_buffer_size: "10mb"
  thread_pool.write.queue_size: "100"
  indices.breaker.total.limit: "60%"
  indices.breaker.fielddata.limit: "15%"
  thread_pool.search.size: "1"
  thread_pool.search.throttled.size: "1"
  thread_pool.index.size: "1"
  thread_pool.write.size: "1"
```

**优化说明**：

| 配置项 | 说明 |
|:-------|:-----|
| `-Xms64m -Xmx160m` | JVM 堆内存（平衡性能和内存） |
| `-XX:+UseG1GC` | 使用 G1GC 替代 SerialGC |
| `bootstrap.memory_lock: "true"` | 启用内存锁 |
| `indices.breaker.total.limit: "60%"` | 断路器防止 OOM |
| IK 分词器 | 已安装 `elasticsearch-analysis-ik-8.13.4` |

> 📎 **IK 分词器安装地址**：https://release.infinilabs.com/analysis-ik/stable/elasticsearch-analysis-ik-8.13.4.zip

## 🚀 第四轮优化：GraalVM JIT 模式迁移 (2026-03-27)

### 迁移目标

将 SpringBoot 后端服务从 OpenJDK/TencentKona JDK 25 迁移到 **GraalVM JDK 25** (JIT 模式)

### 迁移优势

| 优势 | 说明 |
|:-----|:-----|
| ✅ 零代码修改 | 完全兼容现有代码，迁移成本为 0 |
| ✅ 性能提升 | Graal JIT 编译器优化，预期 10-20% 性能提升 |
| ✅ 内存优化 | 更高效的内存管理，预期 5-15% 内存节省 |
| ✅ 启动加速 | 优化的类加载机制，启动时间持平或略快 |
| ✅ 向后兼容 | 完全兼容 HotSpot JVM，无兼容性风险 |

### 实际收益（实测数据）

| 指标 | HotSpot JVM | GraalVM JIT | 改善幅度 |
|:------|:-----------:|:------------:|:--------:|
| **内存占用** | 342.8 MiB | 275.9 MiB | ⬇️ **-19.5%** |
| **堆内存配置** | -Xms64m -Xmx128m | -Xms64m -Xmx128m | 一致 |
| **状态** | 正常运行 | 正常运行 | - |

### 迁移文件

| 文件 | 说明 |
|:-----|:-----|
| `Dockerfile.graalvm` | GraalVM Dockerfile |
| `docker-compose-graalvm.yml` | GraalVM Compose 配置 |
| `deploy-graalvm.sh` | 部署说明文档 |

### 快速迁移命令

```bash
# 1. 停止现有服务
docker stop aurora-springboot && docker rm aurora-springboot

# 2. 启动 GraalVM 服务
docker run -d --name aurora-springboot --network aurora-network -p 8080:8080 \
  -v /opt/aurora/app/aurora-springboot-0.0.1.jar:/app/blog.jar \
  -e SPRING_PROFILES_ACTIVE=prod -e TZ=Asia/Shanghai \
  -e SPRING_DATASOURCE_URL="jdbc:mysql://aurora-mysql:3306/aurora?serverTimezone=Asia/Shanghai&allowMultiQueries=true&useSSL=false&allowPublicKeyRetrieval=true" \
  -e SPRING_DATA_REDIS_HOST=aurora-redis -e SPRING_RABBITMQ_HOST=aurora-rabbitmq \
  -e ELASTICSEARCH_REST_URIS=http://aurora-elasticsearch:9200 \
  -e UPLOAD_MINIO_ENDPOINT=http://aurora-minio:9000/ \
  --restart=always ghcr.io/graalvm/jdk-community:25 \
  java -Xms64m -Xmx128m -XX:+UseSerialGC -XX:MetaspaceSize=64m -XX:MaxMetaspaceSize=128m \
       -XX:ReservedCodeCacheSize=32m -Xss256k -XX:+UseCompressedOops -XX:+UseCompressedClassPointers \
       -XX:+UseStringDeduplication -XX:+OptimizeStringConcat \
       -Djava.security.egd=file:/dev/./urandom -Dspring.jmx.enabled=false -jar /app/blog.jar

# 3. 查看日志
docker logs -f aurora-springboot
```

### JVM 参数对比

| 参数 | HotSpot JVM | GraalVM JIT | 说明 |
|:------|:-----------:|:-----------:|:-----|
| 堆内存 | `-Xms64m -Xmx128m` | `-Xms64m -Xmx128m` | 一致 |
| GC | `-XX:+UseSerialGC` | `-XX:+UseSerialGC` | 一致 |
| 元空间 | `64m-128m` | `64m-128m` | 一致 |
| 代码缓存 | `32m` | `32m` | 一致 |
| **JIT 编译器** | C2 Compiler | **Graal JIT** | ⭐ 核心差异 |

### GraalVM 自动启用的优化

| 优化 | 说明 |
|:-----|:-----|
| ✅ **Graal JIT 编译器** | 替代 C2 编译器，性能提升 10-20% |
| ✅ **更激进的内联** | 方法调用优化，减少 5-10% 开销 |
| ✅ **更好的逃逸分析** | 栈上分配优化，减少堆压力 |
| ✅ **优化的死代码消除** | 减少无效代码，提升 CPU 效率 |

### 注意事项

| 序号 | 注意事项 |
|:----:|:---------|
| 1 | **预热期**：GraalVM 需要运行 10-30 分钟才能体现优化效果 |
| 2 | **镜像说明**：GraalVM 官方没有 Alpine 版本，只有 Oracle Linux |
| 3 | **回滚方案**：保留原配置，可快速回滚到 HotSpot JVM |
| 4 | **生产部署**：建议先在测试环境验证，再灰度发布到生产 |
